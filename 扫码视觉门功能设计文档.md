# 扫码视觉门功能设计文档

## 1. 文档目的

本文档用于定义鞋盒扫码视觉门的功能方案、设备选型、接线方式、控制逻辑和 Python 上位机设计。该视觉门部署在输送线上，用于自动读取鞋盒上的二维码或条形码，并将扫描结果准确关联到对应鞋盒。

本文档基于以下已知条件编制：

- 输送线速度：`0.8 m/s`
- 被扫码对象：鞋盒外箱上的 `1D 条码` 或 `2D 二维码`
- 鞋盒经导向后存在两种稳定姿态：
  - 码面位于鞋盒 `左右侧`
  - 码面位于鞋盒 `前后端`
- 视觉门安装时按现场导向方向 `二选一` 启用：
  - 左右扫码模式
  - 前后扫码模式
- 触发方式：`双光电触发`
- 光电接入：`工控机工业 DI`
- 扫码相机型号：`OPT-IDD1-M050G-L08`

## 2. 设计目标

### 2.1 功能目标

- 在 `0.8 m/s` 输送速度下稳定读取鞋盒条码/二维码
- 基于双光电建立鞋盒跟踪信息，判断扫描结果属于哪个鞋盒
- 支持左右扫码和前后扫码两种安装模式
- 输出 OK/NG 结果给上位系统、PLC 或剔除机构
- 具备日志记录、报警提示、参数配置和追溯能力

### 2.2 设计原则

- 机械结构统一，安装方式兼容两种扫码方向
- 电气接口工业化，光电和执行器均通过工业 DI/O 接入
- 控制逻辑可追溯，扫描结果和鞋盒轨迹强绑定
- Python 上位机以“设备管理 + 轨迹关联 + 日志追溯”为核心，不在工控机上做重图像算法

## 3. 已知设备与参数

### 3.1 扫码相机关键参数

依据用户提供的相机规格书 `扫码相机规格书-OPT-IDD1-M050G-L08-xxx-ESD.pdf`：

- 型号：`OPT-IDD1-M050G-L08`
- 传感器：`CMOS，全局快门`
- 分辨率：`2448 x 2048`
- 最大处理帧率：`30 fps`
- 最大读码速度：`80 个/秒`
- 镜头：`8 mm`
- 读取距离：`50 mm ~ 2000 mm`
- 通信：`TCP / UDP / Serial / Ethernet/IP / FTP / HTTP / Modbus / Profinet / MC / Fins`
- 电气接口：
  - `3 路光耦隔离 DI`
  - `3 路光耦隔离 DO`
  - `1 路 RS232`
  - `1 路 Gigabit Ethernet`
- 供电：`24 VDC ±20%`
- 功耗：`< 30 W`
- 防护：`IP62 / IP65`

| 100 mm | 91.2 x 76.3 mm |

### 3.2 现有安装约束

根据现有草稿和现场意图，先固化以下安装约束：

- 输送机上的鞋盒按中心线居中输送
- 侧面扫码相机与输送方向成一定夹角，以降低镜面反光风险
- 侧面扫码相机与鞋盒最近距离约 `250 mm`
- 上方扫码相机与鞋盒最近距离约 `120 mm`

## 4. 总体方案

### 4.1 系统组成

扫码视觉门由以下部分组成：

- 机械门架
- 扫码相机 `2 台`
- 双光电传感器 `2 只`
- RK3588 工业工控机 `1 台`
- 工业交换机 `1 台`
- 24V 电源系统 `1 套`
- 三色灯/蜂鸣器 `1 套`
- 可选：剔除气缸或 NG 分拣接口

### 4.2 工作模式

#### 模式 A：左右扫码

适用于鞋盒码面位于左右侧面。启用左相机和右相机，对同一个鞋盒左右两侧同时或准同时读取。

#### 模式 B：前后扫码

适用于鞋盒码面位于前后端面。启用前相机和后相机，按照各自安装位置分别触发，对鞋盒前端面和后端面读取。

### 4.3 模式选择原则

- 项目实施时根据导向机构输出的鞋盒姿态选择一种模式
- 默认 `不同时启用左右组和前后组`，避免重复投资和关联复杂度提升
- 机架建议预留两种安装孔位，现场只装对应的一组相机和支架

## 6. 双光电触发方案

### 6.1 双光电作用定义

- `PE1`：检测鞋盒进入视觉门，创建鞋盒跟踪对象
- `PE2`：检测鞋盒到达读码前参考位置，修正速度并生成实际触发时刻

### 6.2 光电安装建议

- 传感器类型优先选 `对射式 24V 工业光电`
- `PE1` 与 `PE2` 沿输送方向布置，推荐间距 `100~150 mm`
- `PE2` 与相机最佳读码平面的距离，建议做成可调，初始建议 `80~120 mm`
- 光电安装高度应与鞋盒稳定遮挡面一致，避免只扫到盒盖或空隙

### 6.3 为什么采用双光电

单光电只能做“到位即拍”，难以处理以下问题：

- 输送速度微小波动
- 鞋盒长短不一
- 连续来料时相邻鞋盒结果容易串箱

双光电可以提供：

- 入站时刻
- 再确认时刻
- 实际线速度估算
- 鞋盒长度估算
- 更稳妥的码-箱绑定窗口

### 6.4 推荐扫码触发策略

本项目建议采用：

- `双光电建轨`
- `相机持续扫码`
- `按鞋盒独立时间窗口绑定码值`

不建议采用“单次触发只扫一帧”的方式，原因是：

- 输送速度为 `0.8 m/s`
- 相机最大处理帧率为 `30 fps`
- 单帧间隔约 `33 ms`
- 鞋盒在单帧间隔内前进约 `26.7 mm`

若只扫单帧，容易因为以下因素造成漏码：

- 反光
- 轻微抖动
- 码只进入部分视野
- 某一帧刚好模糊

因此推荐工作机制为：

- `PE1` 上升沿：创建 `BoxTrack`
- `PE2` 上升沿：修正速度并为当前鞋盒计算读码时间窗口
- 只要系统中存在活动 `BoxTrack`，相机保持持续扫码
- 相机每返回一个码结果，上位机按时间窗口将其绑定到对应鞋盒
- 当所有活动 `BoxTrack` 都结束后，再延时关闭持续扫码

### 6.5 持续扫码的关闭条件

不建议仅按“第一个光电低电平后延时关闭扫码”这一条规则来实现，因为：

- `PE1` 低电平只能表示鞋盒尾部离开 `PE1`
- 它不等于鞋盒已经离开最佳读码区域
- 连续来料时，多个鞋盒可能共用同一段持续扫码时段

因此建议采用以下组合关闭逻辑：

- 单箱时间窗关闭条件：
  - `PE1` 下降沿后加 `tail_delay_ms`
  - 或达到预测离开读码区时间
  - 或达到 `max_track_window_ms`
  - 或读到首个有效码后再保持 `track_hold_after_ok_ms`
- 相机持续扫码关闭条件：
  - 当前活动 `BoxTrack` 数量为 `0`
  - 并超过 `idle_off_delay_ms`

这样可以实现：

- 连续来箱时相机不中断扫码
- 单个鞋盒仍然拥有独立可追溯的读码时间窗
- 码值与鞋盒绑定不会退化成“整段时间只对应一个箱子”

## 7. 关键时序与可行性分析

### 7.1 输送速度分析

- 输送速度：`0.8 m/s = 800 mm/s`
- 若 `PE1` 与 `PE2` 间距为 `100 mm`，则鞋盒通过两传感器的时间约 `125 ms`
- 若 `PE2` 到读码平面距离为 `100 mm`，则从 `PE2` 检测到触发读码还有约 `125 ms`

这意味着：

- Python 上位机通过工业 DI 采集并下发相机触发命令，在时序上是可行的
- 对于更严格的确定性要求，建议预留 `工控机 DO -> 相机 DI` 的硬件触发备份链路

### 7.2 相机能力匹配分析

相机最大读码速度为 `80 个/秒`，远高于本项目常规鞋盒节拍。即使连续来料，系统瓶颈更可能出现在：

- 光照和反光条件
- 码区位置离散度
- 鞋盒间距不足
- 码质量过差

### 7.3 视野风险分析

基于相机规格，`8 mm` 镜头在不同工作距离下的视野固定。需要特别注意：

- 当上方工作距离接近 `120 mm` 时，视野约仅 `100 mm` 量级
- 当侧面工作距离约 `250 mm` 时，视野约在 `200 mm` 量级

因此若鞋盒尺寸较大、码贴位置不固定，可能出现“码在视野外”的风险。建议补充确认：

- 鞋盒最大/最小尺寸
- 码贴相对箱体边缘的位置容差
- 码的最小尺寸和最小模块宽度

如果码位置变化范围较大，应考虑：

- 将相机工作距离提高到 `250~350 mm` 再重新校核视野
- 或更换更短焦距型号
- 或要求导向机构把码区稳定导向到固定窗口

### 7.4 多帧读码可行性分析

若相机按持续扫码方式运行，则上位机不再将“单次触发 = 单帧图像”，而是将其视为：

- 系统进入一个持续读码会话
- 每个鞋盒在会话内拥有独立时间窗
- 相机在窗口期内连续尝试多帧解码

推荐初始参数如下：

- `track_hold_after_ok_ms`：`30~80 ms`
- `tail_delay_ms`：`50~100 ms`
- `idle_off_delay_ms`：`100~200 ms`
- `max_track_window_ms`：`250~500 ms`

在 `0.8 m/s` 线速下：

- `50 ms` 对应位移约 `40 mm`
- `100 ms` 对应位移约 `80 mm`

该量级适合作为读码尾延时和连续扫码关闭延时的初始标定范围。

### 8.2 推荐接线原则

- 光电信号统一接到工控机 `工业 DI`
- 双相机以太网统一通过 `千兆交换机` 接到工控机 `ETH0`
- 工控机 `ETH1` 独立连接调度上位机或上层调度网络
- 工控机 `DO` 负责向外输出 `OK/NG/报警/剔除` 信号
- 若采用 FriendlyElec 普通开发板方案，通常需要额外 `24V -> 12V DC/DC`，且 GPIO 不是工业 24V DI/O

### 8.3 双网口网络拓扑建议

建议固定网络拓扑如下：

- `ETH0`：设备侧专用网口
  - 连接 `千兆工业交换机`
  - 交换机下挂 `Camera A`、`Camera B`
  - 该网段仅用于相机通信
- `ETH1`：上层调度侧网口
  - 连接调度上位机
  - 用于结果上报、参数下发、远程维护或后续与上层系统通讯

建议这样做的原因：

- 相机图像/结果通信与上层调度通信隔离，降低互相影响
- 两台相机只占用工控机一个设备侧网口，布线更清晰
- 便于后续增加维护电脑或扩展第三台以太网设备

推荐初始 IP 规划示例：

| 网口 | 用途 | 示例网段 |
| --- | --- | --- |
| ETH0 | 相机设备网 | `192.168.10.0/24` |
| ETH1 | 调度上位机网 | `192.168.20.0/24` |

设备网示例：

- 工控机 `ETH0`：`192.168.10.10`
- Camera A：`192.168.10.101`
- Camera B：`192.168.10.102`

调度网示例：

- 工控机 `ETH1`：`192.168.20.10`
- 调度上位机：`192.168.20.100`

### 8.4 相机 I/O 预留

相机 `17-pin M12` I/O 可作为预留：

- 备用硬件触发输入
- 读码成功反馈输出
- 联动外部光源或抓拍信号

当前建议默认架构为：

- `PE -> 工控机 DI`
- `工控机 -> 网络命令触发相机`

可选增强架构为：

- `工控机 DO -> 相机 DI0`

这样可以在 Python 完成时序决策后，通过硬件 DO 输出确定性触发脉冲。

若相机支持“触发后持续读码”或“外部使能持续解码”模式，建议优先使用该模式。若相机仅支持单次触发单次返回，则退一步实现为：

- 在单箱有效时间窗内连续触发 `3~5` 次
- 直到读到有效码或窗口超时

但该方式为次优方案，优先级低于真正的持续扫码会话模式。

### 9.2 异常分支

- 相机超时无返回：记 `CAM_TIMEOUT`
- 两台相机都未读到码：记 `NO_READ`
- 两台相机读到不同目标码：记 `AMBIGUOUS`
- 光电顺序异常：记 `SENSOR_SEQUENCE_ERROR`
- 鞋盒间隔过小导致窗口重叠：记 `TRACK_OVERLAP`
- 网络断开或设备离线：记 `DEVICE_OFFLINE`

## 10. 码值与鞋盒关联逻辑

### 10.1 关联核心思路

系统不能只看“哪个相机读到了码”，必须通过 `双光电时序 + 持续扫码会话 + 每箱独立时间窗` 判断码属于哪个鞋盒。

建议采用 `轨迹队列` 模型：

- 每个鞋盒创建一个 `BoxTrack`
- 每个 `BoxTrack` 保存该鞋盒的全部时序信息
- 相机返回的码值只能落入某个 `BoxTrack` 的有效时间窗内
- 系统允许多个 `BoxTrack` 在同一持续扫码会话内并存

### 10.2 BoxTrack 建议字段

```text
track_id
mode                      # LR / FB
pe1_on_ts
pe1_off_ts
pe2_on_ts
pe2_off_ts
speed_mm_s
length_mm
scan_window_start_ts
scan_window_end_ts
scan_close_reason
camera_results[]
final_code
final_status              # OK / NO_READ / AMBIGUOUS / TIMEOUT
```

### 10.3 箱码绑定策略

#### 左右扫码模式

- `Camera_L` 与 `Camera_R` 的结果都绑定到同一个 `BoxTrack`
- 如果两边返回同一个码，直接判定为 `OK`
- 如果只有一边返回有效码，仍可判定 `OK`
- 如果两边返回不同码，判定 `AMBIGUOUS`

#### 前后扫码模式

- 为前相机、后相机分别定义标定偏移量
- 每个相机都绑定到当前 `BoxTrack`
- 若前后结果一致或只有一个有效码，判定 `OK`
- 若结果冲突，判定 `AMBIGUOUS`

### 10.4 防串箱机制

建议同时使用以下约束：

- 基于 `PE1/PE2` 建立先入先出的 `BoxTrack` 队列
- 每个 `BoxTrack` 设置独立的回传有效时窗
- 相机结果必须匹配某个活动时间窗
- 若相邻鞋盒时间窗重叠，则报警并按 `NG` 处理

### 10.5 持续扫码模式下的绑定规则

建议采用以下优先级：

1. 优先按 `camera_id + result_ts` 落入的时间窗匹配
2. 若只命中一个 `BoxTrack`，直接绑定
3. 若命中多个 `BoxTrack`，按与窗口中心距离最小的原则尝试归属
4. 若仍无法唯一判定，则记为 `AMBIGUOUS`
5. 若未命中任何活动窗口，则记为 `UNBOUND_RESULT`

### 10.6 建议增加的可选机制

若现场存在以下情况：

- 输送速度波动大
- 鞋盒间距小于 `30~50 mm`
- 鞋盒长度变化大

建议预留 `编码器` 接口，用输送脉冲代替纯时间估算，可进一步提升码箱关联准确性。

## 11. Python 上位机设计

### 11.1 设计原则

Python 上位机的职责不是识别图像，而是做：

- 设备通信
- 触发时序管理
- 鞋盒轨迹管理
- 扫码结果关联
- 界面显示与日志追溯

因此 RK3588 可以胜任，重点在于 I/O 和网络接口是否工业化。

### 11.2 软件架构

建议采用如下分层：

```text
UI层
  |- 运行监控界面
  |- 参数配置界面
  |- 报警与日志界面

业务层
  |- TrackManager         # 鞋盒轨迹管理
  |- TriggerScheduler     # 触发调度
  |- ResultBinder         # 码值与鞋盒绑定
  |- AlarmManager         # 报警管理

设备层
  |- DiDoService          # 工业DI/O读写
  |- CameraClientA/B      # 相机协议通信
  |- PlcOrMesClient       # PLC / MES通信

存储层
  |- SQLite               # 本地结果与事件库
  |- CSV / JSON 导出
  |- NG图片或相机回传报文归档
```

### 11.3 推荐技术栈

- 系统：`Linux ARM64`
- Python：`3.10+`
- GUI：`PySide6`
- 异步框架：`asyncio`
- 本地数据库：`SQLite`
- 配置：`YAML`
- 日志：`logging + rotating file`
- 服务化：`systemd`

- 主线程：UI

    if track and track.is_complete():
        decision = track_manager.finalize(track)
        io_service.output_result(decision)
        db.save(track)
    if track_manager.no_active_tracks():

## 12. 工控机选型建议

### 12.1 必选条件

工控机建议满足以下最小条件：

- CPU：`RK3588 / RK3588J`
- 内存：`8 GB` 起步
- 存储：`64 GB eMMC` 起步，建议支持 SSD
- 网口：`>= 2 个千兆网口`
- 工业 DI：`>= 2 路`
- 工业 DO：`>= 3 路`
- DI/O 电平：`24V 工业级，优先光耦隔离`
- 电源输入：`9~36VDC` 或直接 `24VDC`
- 工作温度：至少 `0~50°C`，工业环境优先 `-20~60°C` 或更高

### 12.2 对 FriendlyElec RK3588 方案的判断

FriendlyElec 常见 RK3588 平台如 `NanoPC-T6` 提供的是 `3.3V GPIO` 扩展能力，不等同于自带工业 `24V DI/O`。因此：

- 如果项目要求 `工控机必须自带工业 DI/O`，则普通 FriendlyElec RK3588 开发板 `不建议直接作为最终量产工控机`
- 如果已确定使用 FriendlyElec，则需要额外增加：
  - 隔离型 DI 模块
  - 隔离型 DO 模块
  - 24V 转主板工作电压的 DC/DC

这会带来以下影响：

- 接线复杂度增加
- 电气柜空间增加
- 可靠性边界变差
- 与“自带工业 DI/O”的约束不一致

### 12.3 推荐型号

#### 方案 1：Advantech AFE-R761

推荐级别：`优先推荐`

理由：

- RK3588 平台
- 工业级整机设计
- `4 x DI / 4 x DO`，并带隔离
- 多网口、串口、CAN 接口丰富
- 更适合视觉门这种长期在线工业场景

适用结论：

- 满足本项目双光电输入、结果输出和后续扩展要求
- 成本较高，但系统集成风险最低

#### 方案 2：Forlinx FCU3501

推荐级别：`推荐`

理由：

- RK3588J 平台
- 提供 `2 路光耦隔离干接点 DI`
- 提供 `2 路隔离继电器 DO`
- `9~36VDC` 宽压输入
- 工业通讯接口较完整

适用结论：

- 对本项目基础版足够使用
- 若后续需要 `OK / NG / 剔除 / 三色灯` 同时独立输出，建议核对 DO 数量是否够用，必要时外扩

#### 方案 3：JWIPC JEA-E088

推荐级别：`备选，需二次确认`

理由：

- RK3588 平台
- 官方资料显示带 Phoenix 端子 GPIO/IO 资源
- 具备工业形态和较完整的外设

适用结论：

- 需向厂家确认其 `4 in / 4 out` 是否为真正的 `24V 工业 DI/O`、是否隔离
- 若只是 TTL/GPIO 级别，则不建议用于本项目最终电气方案

### 12.4 选型结论

本项目优先建议如下：

1. `首选`：Advantech AFE-R761
2. `性价比方案`：Forlinx FCU3501
3. `已有 FriendlyElec 前提下`：仅建议作为开发验证平台，不建议作为最终量产型工控机，除非厂家可提供自带工业 DI/O 的定制底板

## 19. 机械结构设计建议

### 19.1 门架结构

门架建议采用铝型材或方通焊接结构，形成标准化扫码工位，具备以下特点：

- 横梁高度可调，兼容不同箱高
- 左右相机支架、前后相机支架采用共平台预留孔
- 光电支架可前后微调
- 相机支架带角度刻度，方便现场标定和复位
- 电缆统一走拖链或金属线槽，避免外露摆动

### 19.2 安装调节维度

建议每个相机支架至少具备以下调节能力：

- `X` 向：沿输送方向微调
- `Y` 向：横向微调
- `Z` 向：高度微调
- `Yaw`：相对输送方向水平夹角调节
- `Pitch`：俯仰角调节

### 19.3 结构设计注意点

- 读码区域前后各预留至少 `150 mm` 的无遮挡稳定区
- 避免相机支架直接固定在振动较大的输送机薄板上
- 对于高反光鞋盒，建议预留偏振镜和遮光罩安装位
- 若环境粉尘较大，建议为相机镜头罩预留快速清洁空间

- 班次
- 时间戳
- 轨迹号 `track_id`
- 扫码结果
- 最终码值
- 模式 `LR / FB`
- 处理时长
- 报警代码

### 24.3 通讯建议

- PLC：优先 `Modbus TCP` 或干接点
- MES/WMS：优先 `HTTP API` 或 `MQTT`
- 若现场已有统一协议标准，应优先服从甲方协议

## 25. FAT / SAT 测试清单建议

### 25.1 FAT 测试项目

| 序号 | 测试项目 | 判定标准 |

| 3 | PE1/PE2 触发 | DI 状态与物理动作一致 |

| 5 | 前后模式扫码 | 连续运行成功率满足目标 |

### 25.3 建议验收指标

在未拿到现场样本统计前，可先建议以下占位指标：

- 正常样本读码成功率：`>= 99.5%`
- 连续运行稳定性：`>= 8 小时`
- 码箱绑定正确率：`100%`
- 设备自检耗时：`<= 30 秒`

最终指标应以真实样本测试结果与商务承诺为准。

## 26. 实施与调试建议

### 26.1 实施步骤建议

1. 完成机械门架安装和布线
2. 安装光电并确定 `PE1/PE2` 间距
3. 安装相机并标定最佳工作距离与夹角
4. 接入工控机 DI/O 和网络
5. 完成相机参数、协议和上位机参数配置
6. 使用代表性鞋盒样本做空载调试
7. 使用连续来料做联机调试
8. 确认 OK/NG/剔除联动
9. 导出参数备份并归档

### 26.2 建议先标定的参数

- 线速度补偿系数
- `PE1 -> PE2` 实际间距
- `PE2 -> 相机读码平面` 距离
- 左右相机角度或前后相机基准偏移
- 超时时间
- 最小箱距阈值

## 27. 当前设计假设

本文档当前默认以下假设成立：

- 单条输送线上同一时刻只经过单列鞋盒
- 鞋盒导向后姿态稳定，不会在视觉门内旋转
- 左右模式与前后模式不会在同一班次内频繁切换
- 相机已具备独立读码能力，上位机主要做时序和绑定
- 线体速度标称 `0.8 m/s`，波动范围较小
- 光电安装位置可机械微调

如果这些假设后续不成立，则需要同步调整：

- 光电布置
- 软件绑定策略
- 相机镜头/工作距离
- 工控机 I/O 数量
- 是否增加编码器和缓存段
